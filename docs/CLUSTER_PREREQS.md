# Cluster Prerequisites for AuthorWorks

This document records the cluster-wide pieces AuthorWorks's GitOps depends on.
As of 2026-04-28 every item in sections 1-5 and 7 is **DONE** on the homelab
cluster (`l3ocifer/homelab` repo). The only remaining prerequisite is
**section 3 (`ghcr-credentials` Secret)**, which is per-cluster bootstrap.

Once 3 is satisfied and `scripts/seal-secrets.sh` has been run, the AuthorWorks
deploy is a single `kubectl apply -f k8s/argocd/app-of-apps.yaml`.

## 1. ArgoCD core — DONE

| Resource | Namespace | Notes |
|----------|-----------|-------|
| `argocd-server` Deployment | `argocd` | Listens on `:80` plaintext (gRPC + HTTP). |
| `argocd-application-controller` StatefulSet | `argocd` | |
| `argocd-applicationset-controller` Deployment | `argocd` | |
| `argocd-image-updater` Deployment | `argocd` | See #2. |
| `argocd-repo-server` Deployment | `argocd` | |
| AppProject `homelab` | `argocd` | Permissive (`sourceRepos: '*'`, `destinations: '*'`). The AuthorWorks ApplicationSet uses this project. |

## 2. argocd-image-updater configuration — DONE

The image-updater Deployment in `infrastructure/argocd-image-updater.yaml`
runs `quay.io/argoprojlabs/argocd-image-updater:v0.14.0` with
`applications_api: kubernetes`. In this mode the updater reads/writes
`Application` CRDs through the Kubernetes API directly and never opens a
gRPC connection to `argocd-server`, which sidesteps the long-standing
"https on port 80 / TLS verify fails" bug in v0.14.0. Confirmed working:

```bash
kubectl logs -n argocd deploy/argocd-image-updater --tail=20
# "ArgoCD configuration: [apiKind=kubernetes, ...]"
# periodic "Starting image update cycle" lines, no TLS errors
```

The earlier upgrade attempt to `v1.0.4` was reverted because that version
is a controller-runtime rewrite that requires a new `ImageUpdater` CRD per
application. Stay on `v0.14.0` + `applications_api: kubernetes` until/unless
the homelab decides to migrate everyone to the v1.x model.

The Service Account needs read/write on `applications.argoproj.io` in the
`argocd` namespace; that's already granted in `infrastructure/argocd-image-updater.yaml`.

## 3. GHCR pull secret for image-updater — REQUIRED

`argocd-image-updater` reads the GHCR registry API to discover new tags.
The annotation on the AuthorWorks Application points at:

```
argocd-image-updater.argoproj.io/pull-secret: pullsecret:argocd/ghcr-credentials
```

So a Docker config Secret must exist in the `argocd` namespace:

```bash
kubectl create secret docker-registry ghcr-credentials \
  --namespace argocd \
  --docker-server=ghcr.io \
  --docker-username=<github-user> \
  --docker-password=<PAT-with-read:packages>
```

The PAT only needs `read:packages`. Reuse it for the per-app
`ghcr-pull-secret` if convenient (the AuthorWorks app's `ghcr-pull-secret`
lives in the `authorworks` namespace, sealed via SealedSecrets — that's
separate from this one).

## 4. sealed-secrets controller — DONE

Running in `kube-system` (`sealed-secrets-controller`). Active key
auto-discovered by `kubeseal --fetch-cert`. The helper at
`scripts/seal-secrets.sh` re-seals the AuthorWorks SealedSecrets against
this controller's current key.

## 5. cert-manager + Traefik — DONE

The `authorworks-ingress` references:
- `cert-manager.io/cluster-issuer: "letsencrypt-prod"`
- `ingressClassName: traefik`

Both present cluster-wide.

## 7. ApplicationSet ownership conflict — DONE

The external `production-apps` ApplicationSet in `l3ocifer/homelab`
(`argocd/apps/production-apps.yaml`) used to generate
`Application/authorworks` with malformed image-list annotations
(`app={{image}}:latest{{extraImages}}` with empty `extraImages`, only
tracking one image, never matching the immutable `sha-*` tags this repo
publishes).

That entry has been commented out (commit `775e469` in `l3ocifer/homelab`),
so:

- `Application/authorworks` was garbage-collected by ArgoCD's
  `resources-finalizer`.
- `Application/authorworks-homelab` (generated by THIS repo's ApplicationSet
  at `k8s/argocd/applicationset.yaml`) is now the sole owner of the
  workloads in the `authorworks` namespace.

Re-enabling the entry in the ops repo without removing the AuthorWorks
ApplicationSet first would re-create the conflict.

## 8. Verifying after prereqs are in place

```bash
# kubectl context pointed at the cluster
./scripts/bootstrap-argocd.sh
kubectl -n argocd get application authorworks-homelab        # Synced + Healthy
kubectl -n authorworks get pods                              # all Running
kubectl -n argocd logs deploy/argocd-image-updater --tail=50 # no errors; "Processing results: ... images_updated=N"
```
