# Homelab Setup Checklist (k3s)

End-to-end checklist for getting AuthorWorks **synced and healthy** on the
cluster: login, create/edit books, AI outline + full-book generation. Cluster
target: `ssh alef`.

The deploy itself is fully GitOps now — ArgoCD owns sync, and ArgoCD Image
Updater bumps image tags automatically when CI publishes a new
`sha-<commit>`. The steps below are bootstrap + first-time data work.

---

## 0. Cluster prerequisites

These items are managed cluster-wide (in `l3ocifer/homelab`). As of
2026-04-28 every item below is **DONE** except 0.3 (`ghcr-credentials`
Secret in the `argocd` ns), which is per-cluster bootstrap. Full handoff
version: [`CLUSTER_PREREQS.md`](./CLUSTER_PREREQS.md).

### 0.2 argocd-image-updater configuration — DONE

Runs `quay.io/argoprojlabs/argocd-image-updater:v0.14.0` with
`applications_api: kubernetes` (in `l3ocifer/homelab`'s
`infrastructure/argocd-image-updater.yaml`). In this mode the updater
reads/writes `Application` CRDs through the Kubernetes API directly and
never opens a gRPC connection to `argocd-server`, sidestepping the
long-standing v0.14.0 bug where `--argocd-plaintext` was acknowledged in
the startup log but the gRPC client still emitted `https://...` URLs.

Verify:

```bash
kubectl logs -n argocd deploy/argocd-image-updater --tail=20
# "ArgoCD configuration: [apiKind=kubernetes, ...]"
# "Starting image update cycle, considering N annotated application(s)"
# No x509 / "certificate signed by unknown authority" lines.
```

The earlier upgrade attempt to `v1.0.4` was reverted because that version
is a controller-runtime rewrite that requires a new `ImageUpdater` CRD
per Application — too disruptive to roll out cluster-wide.

### 0.3 GHCR credentials for image-updater — REQUIRED

The ApplicationSet annotation
`argocd-image-updater.argoproj.io/pull-secret: pullsecret:argocd/ghcr-credentials`
points at a dockerconfigjson Secret in the `argocd` namespace. Create it
with a PAT that has `read:packages`:

```bash
kubectl create secret docker-registry ghcr-credentials \
  --namespace argocd \
  --docker-server=ghcr.io \
  --docker-username=<github-user> \
  --docker-password=<PAT>
```

This is **separate** from `authorworks/ghcr-pull-secret` (which the
workload Pods use to pull images). Same PAT is fine for both.

### 0.4 sealed-secrets controller — DONE

Already running in `kube-system` as `sealed-secrets-controller`. The
homelab `sealed-secrets.yaml` and `ghcr-sealed-secret.yaml` in this repo
must be re-sealed against the controller's current key — see Section 2.

### 0.5 cert-manager + Traefik (for the Ingress) — DONE

The `authorworks-ingress` references `cert-manager.io/cluster-issuer:
"letsencrypt-prod"` and `ingressClassName: traefik`. Both are cluster-wide.

### 0.7 ApplicationSet ownership conflict — DONE

The external `production-apps` ApplicationSet in `l3ocifer/homelab`
(`argocd/apps/production-apps.yaml`) used to generate
`Application/authorworks` with a malformed `image-list`
(`app={{image}}:latest{{extraImages}}` with empty `extraImages`, only
tracking one image at a tag this repo no longer publishes).

That entry is now commented out (commit `775e469` in `l3ocifer/homelab`),
so:

- `Application/authorworks` was GC'd by ArgoCD's `resources-finalizer`.
- `Application/authorworks-homelab` (generated by this repo's
  ApplicationSet) is now the sole owner of workloads in the `authorworks`
  namespace.

Re-enabling the entry in the ops repo without removing the AuthorWorks
ApplicationSet first would re-create the conflict.

## 1. Database migrations (order matters)

Run against cluster PostgreSQL (`postgres.databases.svc.cluster.local:5432` or
a port-forward):

1. **Base schema** — creates `content` (and optionally `public`) schema/tables. Use your existing init or `scripts/schema.sql` if present.
2. **Credit system** — `psql $DATABASE_URL -f scripts/migrations/001_add_credit_system.sql`
3. **Frontend app tables** — `psql $DATABASE_URL -f scripts/migrations/002_frontend_schema.sql`

**Verify:** `content.books`, `content.chapters`, `content.generation_jobs`
exist (and `public.generation_logs` if Logto/legacy was used).

## 2. Re-seal SealedSecrets and bootstrap ArgoCD

The `sealed-secrets.yaml` and `ghcr-sealed-secret.yaml` checked into this
repo were sealed against an older controller key and **cannot be decrypted on
the new cluster**. Re-seal them:

```bash
./scripts/seal-secrets.sh
# Provide plaintext via env vars, or interactive prompts:
#   GHCR_USERNAME, GHCR_TOKEN, DATABASE_URL, LOGTO_DATABASE_URL, REDIS_URL,
#   LOGTO_APP_ID, LOGTO_APP_SECRET, JWT_SECRET (or "generate"), ANTHROPIC_API_KEY
```

This overwrites both YAML files in `k8s/overlays/homelab/` with values that
the current cluster's controller can decrypt. Commit and push.

Then bootstrap ArgoCD:

```bash
./scripts/bootstrap-argocd.sh
# Pre-flights cluster, optionally re-runs seal step, then:
#   kubectl apply -f k8s/argocd/app-of-apps.yaml
#   waits for Application authorworks-homelab to be Healthy
```

Expected resources after bootstrap:

```
argocd:        Application authorworks-apps (root)
               ApplicationSet authorworks
               Application authorworks-homelab
authorworks:   Deployments  authorworks-frontend, authorworks-book-generator, authorworks-content-worker
               Services     authorworks-frontend, authorworks-book-generator
               Ingress      authorworks-ingress -> author.works
               Secrets      authorworks-secrets, ghcr-pull-secret  (decrypted from SealedSecrets)
```

## 3. CI and image flow

GitHub Actions (`.github/workflows/docker-homelab.yml`) builds three images on
every push to `main`:

| Image | Path |
|-------|------|
| `ghcr.io/authorworks/frontend` | `frontend/app/` |
| `ghcr.io/authorworks/book-generator` | `core/book-generator/` |
| `ghcr.io/authorworks/content-worker` | `workers/content/` |

Tags pushed: `sha-<7chars>` (immutable, what image-updater watches) and
`latest` (human reference, ignored by image-updater).

Once the workflow finishes, ArgoCD Image Updater (next two-minute cycle)
detects the new SHA, patches `Application/authorworks-homelab` with
`spec.source.kustomize.images[*].newTag = sha-<new>`, and ArgoCD rolls the
deployments forward. **There is no SSH `kubectl rollout restart` step** —
that was retired with the old workflow.

To verify a new build flowed all the way through:

```bash
gh run watch                                                    # CI green
kubectl logs -n argocd deploy/argocd-image-updater --tail=50    # bump line
kubectl get application authorworks-homelab -n argocd -o jsonpath='{.spec.source.kustomize.images}'
kubectl rollout status deployment/authorworks-frontend -n authorworks
```
